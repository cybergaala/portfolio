When Swift concurrency landed it felt too good to be true. No more callback pyramids. No `DispatchQueue.main.async` scattered everywhere. Structured concurrency with `async/await` that the compiler actually understands.

Two years in, it mostly delivered. But there are edges worth knowing before you go all in.

## The good

`async/await` is genuinely better for sequential async work. A network call that fetches a user, then their posts, then their media — that's three awaits on three lines. The old version was three nested closures, three levels of indentation, and three places to forget to call the completion handler.

Actors solved a real problem cleanly. I used to have `DispatchQueue` properties on every class that touched shared state. Now I mark the class `actor` and the compiler enforces isolation. The data races that used to show up in production after two months of use just don't happen.

`Task` groups let you run concurrent work and collect results without building your own machinery. Fetching five things in parallel and waiting for all of them is ten lines, readable ones.

## The rough edges

The main gotcha is crossing actor boundaries. Every `await` is a suspension point, and the world can change while you're suspended. If you load data, suspend to hit the network, and come back, the view hierarchy might have changed under you. This is correct behavior — but it takes time to internalize.

`@MainActor` helps. Marking a ViewModel `@MainActor` means all its mutations happen on the main thread and you don't think about it. But it also means every call to that actor from background work is an implicit hop, and if you're not careful you end up with more context switching than you had with explicit queues.

## Where I still use completion handlers

Third-party SDKs that haven't migrated yet. Any time I'm wrapping a delegate-based API. And anywhere I need cancellation semantics that predate the `Task` cancellation model.

The rule I follow: new code is always `async/await`. Existing code gets migrated when I'm already touching it for another reason.

## The bottom line

Swift concurrency is the right direction. The learning curve is real but it's a one-time cost. Write some actors, introduce a TaskGroup, get burned by one actor isolation bug — then you understand the model and the rest clicks.
