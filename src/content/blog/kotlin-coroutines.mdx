Coroutines are one of those APIs that feel like magic until they don't. You write `launch` and `await` and things work. Then something behaves unexpectedly and you realize you never understood the model.

This is my attempt to explain it without magic.

## What a coroutine actually is

A coroutine is a unit of work that can be suspended and resumed. That's it. When you `suspend` inside a coroutine, you give up the thread. The thread goes off to do other work. When the suspension is done, the coroutine resumes — possibly on a different thread.

This is different from blocking. A blocking call holds the thread hostage. A suspending call releases it.

```kotlin
// Blocking — thread is stuck until the network responds
val user = userApi.fetchUserBlocking(id)

// Suspending — thread is free while we wait
val user = userApi.fetchUser(id)
```

The compiler transforms the second one into a state machine. Every `suspend` call becomes a state transition. The runtime knows how to resume from each state.

## Dispatchers

A `Dispatcher` controls which thread (or thread pool) runs your coroutine.

- `Dispatchers.Main` — the Android main thread. Use for UI work.
- `Dispatchers.IO` — a pool sized for blocking I/O. Use for network and disk.
- `Dispatchers.Default` — a pool sized for CPU work. Use for parsing and computation.

The common mistake is running database or network work on `Main` and wondering why the UI stutters. Move it to `IO` with `withContext`.

```kotlin
suspend fun loadUser(id: String): User = withContext(Dispatchers.IO) {
  database.getUser(id)
}
```

## Scope

Every coroutine runs in a `CoroutineScope`. The scope controls the coroutine's lifetime. When the scope is cancelled, all coroutines in it are cancelled.

In Android, `viewModelScope` is your best friend. It's tied to the ViewModel's lifecycle and cancels automatically when the ViewModel is cleared.

```kotlin
viewModelScope.launch {
  val user = userRepository.loadUser(id)
  _uiState.value = UiState.Success(user)
}
```

Don't create `GlobalScope.launch` in production code. It has no lifecycle and leaks easily.

## Structured concurrency

The rule of structured concurrency: a coroutine's lifetime is bounded by its scope. Child coroutines are scoped to their parent. If the parent is cancelled, the children are cancelled.

This makes cancellation predictable. You cancel the scope at the right place in the lifecycle and everything cleans up.

```kotlin
// Both run concurrently; both are cancelled if the scope is cancelled
coroutineScope {
  launch { loadPosts() }
  launch { loadComments() }
}
```

## The mental model

Think of coroutines as lightweight threads that share threads efficiently. The suspension mechanism is cooperative — coroutines yield control explicitly. The scope hierarchy makes cancellation deterministic. The dispatcher controls where things run.

Once those three concepts are clear, the API stops feeling like magic and starts feeling like a well-designed system.
